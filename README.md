# Auto-LifeOS Public Demo

> AI Agent 产品案例：把“今天吃什么”的自然语言需求，转成经过检索、库存与饮食规则校验、双阶段确认和事务更新的可执行流程。

- 在线 Demo：[https://auto-lifeos-demo.streamlit.app/](https://auto-lifeos-demo.streamlit.app/)
- GitHub：[https://github.com/lincoln3079938241-coder/auto-lifeos-demo](https://github.com/lincoln3079938241-coder/auto-lifeos-demo)
- 作者：陈蔚昕（Weixin Chen）

## 产品问题与解决方案

**用户问题：** 日常饮食建议常与家中真实食材脱节。用户得到一道菜后，仍要自行核对库存、过期状态、忌口和实际使用量；文字建议也不应直接触发库存副作用。

**产品目标：** 让用户用自然语言表达时间、口味或饮食要求，系统结合演示档案、检索证据和当前库存生成结构化方案；只有经过规则校验和两次明确确认，才更新库存。

**核心价值：**

- 建议落到当前演示库存与可执行菜谱；
- 模型/Mock 负责提出方案，确定性规则负责安全与可执行性；
- 双阶段确认隔离“接受建议”和“执行副作用”；
- 事务、审计和撤销让结果可追踪、可回退。

当前 Demo 包含 46 种内置食材、100+ 条中英文日常别名、20 项默认库存和 28 道结构化菜谱。所有内容均为 synthetic/sample，不构成医疗建议。

## Agent 架构

```text
用户需求输入
  → 意图理解
  → 档案 / TF-IDF 知识 / 库存读取
  → 方案生成（公开版强制 Mock）
  → Pydantic 合约 + 库存 / 过敏 / 忌口 / 过期 / 单位规则
  → 用户第一次确认
  → 实际用量与第二次确认
  → SQLite 原子库存事务
  → 饮食记录 / 审计 / 撤销与异常回退
```

LangGraph 负责节点编排、条件路由、重试和回退；Provider 不直接写库存。公开版不读取 API Key，也不声称正在调用 GPT、Qwen 或其他真实模型。

## 第一次使用

1. 直接输入想吃什么，或点击任意示例。
2. 点击“帮我推荐”。
3. 查看菜名、时间、营养、食材和简单做法。
4. 点击“这个方案可以”，再核对实际使用量。
5. 第二次确认后查看演示库存变化，需要时可以撤销。

无需登录、无需 API Key。所有用户、食材、饮食习惯和实验数据都是 synthetic/sample，不构成医疗建议。

如果想先调整食材，可以点击首页的“先调整演示库存”。不调整也可以直接使用 20 项基础库存完成原有流程。

## 日常食材与菜谱

内置目录覆盖主食、肉蛋奶、豆制品、蔬菜、水果和基础调味料。28 道菜谱分为：

- 早餐 6 道
- 午餐/晚餐 18 道
- 轻食 4 道

菜谱保存在 `data/sample_recipes.json`，每道菜都有独立的食材、用量、步骤、标签、替代规则和演示营养估算。首页仍只显示一个主推荐，不会一次展示完整菜谱库。

Mock Provider 不使用自由生成的固定话术，而是对可执行菜谱做确定性排序：

- 主要食材库存覆盖
- 早餐、午餐、晚餐或轻食场景匹配
- 烹饪时间硬过滤
- 高蛋白、低脂、素食等标签匹配
- 演示偏好匹配
- 即将过期食材利用
- 用户明确提到的食材匹配
- 安全替代产生的小幅惩罚

过敏、忌口、过期、单位不兼容和主要食材库存不足会先于评分淘汰菜谱。页面只展示普通中文推荐原因，不显示原始分数。

有限替代规则包括鸡胸肉/鸡腿肉、牛肉/瘦猪肉、西兰花/青菜、大米/面条、早餐中的全麦面包/燕麦、牛奶/无糖豆浆、虾仁/鱼肉等。替代后会重新检查库存、单位、过敏、忌口和营养估算，并在推荐卡片中明确说明。

## 公开演示边界

- 强制使用 Mock Provider，不连接外部模型，不需要任何 API 凭据。
- 所有用户、库存、知识库与 A/B 数据均为 synthetic/sample。
- 每个 Streamlit 浏览会话使用独立的临时 SQLite 数据库；新会话恢复相同基线。
- 数据只用于软件演示，服务重启后可以重置，不构成医疗建议。
- 项目不包含真实用户数据、本地数据库、求职资料或主项目 Git 历史。

## 编辑演示库存

“演示库存”页面支持搜索、新增、修改数量与单位、修改保质期与存放位置、删除，以及恢复当前会话的初始库存。

- 每个新会话从 20 项 synthetic/sample 基础库存开始。
- “西红柿”“鸡胸”“燕麦片”等别名会映射到已有标准食材。
- 添加重复食材时不会生成重复记录，用户可以确认合并数量或取消。
- 未收录的新食材会获得当前会话专属 ID，并标记“营养数据暂缺”；系统不会编造营养数值，也不会自动编造新菜谱。
- 删除采用软删除：食材不再参与推荐，但历史饮食事务仍可读取。
- 新增、合并、修改、删除和恢复都会写入当前会话审计记录。
- 推荐每次都读取修改后的有效库存；过期、数量不足或已删除食材不能进入可执行计划。

## 功能

- 普通中文引导、三个一键示例与手机友好的四步流程
- 会话级演示库存搜索、新增、修改、软删除、恢复与审计
- 46 种内置食材、28 道结构化日常菜谱与可解释确定性排序
- 基础调味料简化检查与有限安全替代
- 智能饮食推荐与 Pydantic 结构化计划
- 库存、过敏原、过期、单位与营养范围的确定性校验
- 方案确认、实际用量修改、实际使用量确认
- 原子库存扣减、前后对比与当前会话撤销
- TF-IDF 私域知识检索
- 可读的 LangGraph Agent 执行轨迹
- 两种 Prompt 各 20 个合成案例的对比实验
- 会话隔离、架构与安全说明

## 技术栈

- 产品界面：Streamlit

- 测试：pytest + Streamlit AppTest

## 产品限制

- 公开版强制使用 Mock Provider，不代表真实 LLM 的推理质量；
- 所有用户、库存、知识、菜谱营养值和实验案例均为 synthetic/sample；
- 单位换算、包装规格和安全替代规则覆盖有限；
- SQLite 与临时会话数据库用于演示，不代表生产级多用户架构；
- 合成 Prompt 对比不是用户实验，不提供满意度、留存、转化或医疗效果结论。
